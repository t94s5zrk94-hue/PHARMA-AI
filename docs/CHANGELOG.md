# Versioning Policy

Pharma AI follows Semantic Versioning.

```
MAJOR.MINOR.PATCH
```

Meaning

MAJOR

Breaking architectural changes.

Example

```
1.0.0

↓

2.0.0
```

---

MINOR

New functionality.

Example

```
1.4.0

↓

1.5.0
```

---

PATCH

Bug fixes.

Example

```
1.5.0

↓

1.5.1
```

# Known Issues Policy

Known issues should be documented openly.

Each issue should include:

- Description
- Severity
- Affected Module

- Planned Resolution

---

# Issue Severity

Recommended severity levels:

| Severity | Meaning |
|----------|---------|
| Critical | Immediate fix required |
| High | Major functionality affected |
| Medium | Partial functionality affected |
| Low | Minor inconvenience |
| Informational | No functional impact |

Severity should be determined consistently.

---

# Maintenance Releases

Maintenance releases focus on stability.

Typical contents include:

- Bug fixes
- Performance improvements
- Documentation corrections
- Validation improvements

Maintenance releases should avoid major architectural changes.

---

# Hotfix Policy

Hotfix releases should be reserved for production-critical issues.

Typical examples:

- Data corruption
- Critical validation failure
- Security issue
- Clinical logic defect

Hotfixes should follow an accelerated governance process while preserving validation and audit requirements.

---

# Deprecation Policy

Deprecated functionality should not be removed immediately.

Recommended lifecycle:

```text
Supported

↓

Deprecated

↓

Replacement Available

↓

Removal
```

Deprecation should be documented before removal.

---

# Backward Compatibility

Minor and patch releases should preserve backward compatibility whenever practical.

Breaking changes should normally occur only in major releases.

When backward compatibility cannot be maintained:

- Document the change
- Explain the impact
- Provide migration guidance

---
